<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions → npm registry | Publish identity (OIDC token or NPM_TOKEN) crosses here |
| repo working tree → published tarball | `files` allow-list controls leaked contents |
| repo metadata → external registries | server.json / marketplace listing claim an identity + a working server |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-03 | Spoofing/Elevation | npm publish auth | mitigate | Prefer OIDC trusted publishing (no stored long-lived token); `id-token: write` scoped to the publish job only; trusted-publisher filter pins repo+workflow (V2/V4) |
| T-08-13 | Elevation | compromised CI step exfiltrating OIDC token | mitigate | Minimal publish-job steps, pinned action versions (`@v6`/`@v5`), `id-token: write` not granted to other jobs |
| T-08-02 | Information Disclosure | over-broad published tarball | mitigate | `files: ["dist"]` (Plan 01) + `npm pack --dry-run` review in the checkpoint; Gitleaks gate in security.yml backstops |
| T-08-12 | Repudiation | advertising a non-functional MCP server | mitigate | Checkpoint requires confirming Plan 01's stdio-server.spec passed before any registry/marketplace submission |
| T-08-SC3 | Tampering | mcp-publisher tooling at publish time | accept | mcp-publisher is official MCP tooling installed at release; submission is a human/CI step gated by the checkpoint |
</threat_model>
