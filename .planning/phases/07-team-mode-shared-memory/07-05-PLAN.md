```typescript
export const retrieveMemoriesResponseSchema = z.object({
  ok: z.literal(true),
  memories: z.array(retrievedMemorySchema),
  total: z.number().int().nonnegative(),
  // Pre-rendered startup-injection block (CR-01): includes the `author:`
  // prefix annotation for teammate-authored memories.
  startupInjection: z.string(),
});
```

Current search.ts (src/cli/commands/search.ts:18-38) — after the `!result.ok` guard it does
ONLY `console.log(formatTable(result.memories));` and never reads `result.startupInjection`.

formatStartupInjection (src/core/injection/formatStartupInjection.ts) renders a block whose
header is the literal string `"Relevant prior context"`, with one `- [<kind>] <author>: <content> (...)`
line per included memory. The `author: ` prefix appears ONLY when the memory's `author` is
non-empty AND the service's `localUsername` is set AND `author !== localUsername` (D-10).
`memoryCoreService.retrieveMemories` calls it with `localUsername: localAuthor` (the service's
resolved username), so the prefix depends on the SERVICE username, not the seeding username.

Test helpers (tests/helpers/cliTestContext.ts):
- `createTestCliContext()` → seeds three memories (author stamped to the default resolved
  username) and returns `{ db, service, projectId, dbPath, cleanup }`.
- `createTeamUserContext({ username, projectId? })` → empty DB, service built with an explicit
  `username`, so memories stored through it are stamped `author = <username>`.
- `createMemoryCoreService({ db, username })` (src/core/api/memoryCoreService.ts) — a second
  service can be constructed over the SAME `db` with a DIFFERENT `username`; that is how a
  test makes the local username differ from a memory's stored author.
